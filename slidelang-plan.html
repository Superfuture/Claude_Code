Section 01

TL;DR

A hosted deck-as-code authoring platform. Write or generate a strict YAML deck spec; a deterministic compiler renders it to editable HTML slides; a validator measures real DOM boxes and flags overflow, collisions, contrast, and chart-sanity issues; one click runs an LLM-assisted repair that emits a YAML diff. The same compiler runs from a CLI.

Wedge. Prompt-to-slides tools (Gamma, Tome) produce static, brittle output. Slidelang treats the deck as a typed program — every block is positioned, validatable, diffable — so AI and humans collaborate on the same source of truth.

Section 03

Deck spec schema

Percent-based coordinates mean validator and renderer share one geometry. Overflow is just x + w > 100.

# deck.yaml
meta:
  title: Q3 Board Update
  theme: light            # light | dark
  size: { w: 1280, h: 720 }
slides:
  - id: title
    blocks:
      - { type: text, x: 8, y: 35, w: 84, h: 20,
          content: "Q3 Board Update", style: title }
      - { type: text, x: 8, y: 58, w: 84, h: 8,
          content: "Acme Inc. — October 2026", style: subtitle }

  - id: metrics
    blocks:
      - { type: text,  x: 6, y: 6,  w: 88, h: 10, content: "Quarterly Metrics", style: h1 }
      - { type: chart, x: 6, y: 20, w: 56, h: 70,
          spec: { $schema: "https://vega.github.io/schema/vega-lite/v5.json",
                  data: { values: [...] }, mark: bar, encoding: {...} } }
      - { type: text,  x: 66, y: 22, w: 28, h: 60, content: "ARR up 38%..." }

  - id: math
    blocks:
      - { type: math,  x: 10, y: 30, w: 80, h: 30,
          latex: "\\text{LTV} = \\frac{\\text{ARPU}}{\\text{churn}}" }

  - id: image
    blocks:
      - { type: image, x: 0, y: 0, w: 100, h: 100,
          source: { provider: flux, prompt: "minimalist data visualization, dark mode" } }
      # or { provider: unsplash, query: "office team" }
      # or { provider: url, src: "https://..." }

Validation invariants

  • 0 ≤ x, y and x + w ≤ 100 and y + h ≤ 100
  • Pairwise non-overlap among blocks on the same slide (allow-list for intentional layers)
  • Rendered text height ≤ block height (measured in a hidden ghost DOM)
  • WCAG AA contrast for any text block over a colored or image background
  • Chart sanity: encodings present, axis fields exist in data.values[0], units consistent
Section 04

System architecture

One TypeScript compiler module, imported by the browser and by the CLI. That single fact is what makes "deck-as-code" real and gives the TDD a clean diagram.

┌─────────────────────────────────────────────────────────┐ │ Browser (Vite + React) │ │ ┌──────────────┐ ┌───────────────────────────────┐ │ │ │ Monaco YAML │ ⇄ │ Compiler (TS, isomorphic) │ │ │ └──────────────┘ │ yaml → AST → React vDOM │ │ │ └───────────────────────────────┘ │ │ ↓ │ │ ┌───────────────────┐ │ │ │ Slide canvas │ │ │ │ + Validator │ ← runs in DOM │ │ │ + Repair pills │ │ │ └───────────────────┘ │ └──────────────────────────┬──────────────────────────────┘ │ fetch ▼ ┌─────────────────────────────────────────────────────────┐ │ Cloudflare Worker (Hono) │ │ POST /generate → Claude Opus 4.7 (structured out) │ │ POST /repair → Claude Opus 4.7 (constrained patch) │ │ POST /image → Flux via Replicate (cached in R2) │ │ GET/PUT /d/:id → KV (deck spec by share id) │ └─────────────────────────────────────────────────────────┘ CLI (Node): slidelang gen "prompt" → calls /generate, prints YAML slidelang render deck.yaml → emits standalone deck.html
Section 05

AI generation pipeline

  1. Plan. Opus 4.7 receives the prompt + JSON Schema, emits an outline (slide titles + intents). Streamed to UI as a filmstrip skeleton.
  2. Draft per slide. Parallel calls, one per slide, each constrained by the block JSON Schema. Provisional positions included.
  3. Compile + validate. Local, in-browser, ~10 ms.
  4. Auto-repair pass. For each flagged issue, send {slide_spec, issue} to /repair. System prompt: "Return only the patched blocks as JSON." Apply diff, re-validate, ≤2 retries.
Opus 4.7 is the right call for planner/critic. Worker calls cache the system prompt + schema via Anthropic prompt caching, so repeated /repair requests are cheap and fast.
Section 06 — the showpiece

Validator + repair detail

This is the section that has to land on video. Everything in the demo arc leans on it.

Rule engine (~200 LOC)

type Issue =
  | { kind: 'overflow';  slideId; blockId; measured: number; bound: number }
  | { kind: 'collision'; slideId; aId; bId; overlapPct: number }
  | { kind: 'contrast';  slideId; blockId; ratio: number; required: 4.5 }
  | { kind: 'chart';     slideId; blockId; reason: string };

UI surface

Each issue renders as a red pill anchored over the offending block (position: absolute). Clicking opens a popover with:

  • Plain-English description
  • 3-line YAML diff preview (red / green)
  • Repair button → applies patch + re-validates
  • Dismiss → keep the issue, hide the pill
In the demo I deliberately overflow a bullet list. Red pill appears in <100 ms (deterministic check). Click Repair. Opus returns { font_size: 22, content: "<tightened copy>" }. Diff applies. Pill clears. Total: ~4 seconds of pure depth-area magic.
Section 07

PRD & TDD outlines

PRD — 1 to 2 pages

  1. Problem. Prompt-to-slides tools produce brittle, static output. Editing feels like fighting the AI. Teams revert to PowerPoint.
  2. Target user. Operators, founders, and engineering leads writing recurring business or technical decks.
  3. Why deck-as-code. Typed source means AI and humans edit the same artifact — diffable, reviewable, versionable, programmable.
  4. The trust loop. Generation is cheap; trustable generation is the wedge. The validator + repair loop turns AI output from a starting point into something shippable.
  5. Wedge → platform. Start with board updates and technical reviews. Expand to agent-authored decks, design-system templates, embeddable charts, programmatic decks from data sources.
  6. Non-goals (v0). Real-time collaboration, PPTX export, branded templates, mobile authoring.
  7. Success metric. Time-to-shippable-deck. Target: 3 minutes from prompt to share link.

TDD — 1 to 2 pages

  1. Architecture diagram (browser ⇄ Worker ⇄ KV/R2 + shared compiler + CLI).
  2. Deck spec schema. One source of truth: Zod → JSON Schema → TS types.
  3. Compiler: pure (spec) → React tree. Isomorphic, runs in browser and Node.
  4. AI planning. Outline-then-draft, per-slide parallel structured calls. Prompt caching.
  5. Browser editor state model. YAML text → debounced parse → spec → render. Repair patches mutate the YAML AST.
  6. Validation + repair pipeline. Deterministic rules over the DOM; LLM critic only for content rewrites. Two-retry budget.
  7. Chart / math / image rendering. Why Vega-Lite (declarative). Why KaTeX (sync). Image provider abstraction with R2 cache.
  8. Hosted API. Workers, Hono, KV, R2. Edge latency targets.
  9. Publishing flow. /d/:id is KV → static render. No JS needed to view.
  10. CLI / agent surface. Same compiler imported as @slidelang/compiler. Plugin hook sketch for v0.2.
  11. What I'd do with more time. Real-time collab via Y.js, theme system, slide-level iteration, agent workflows.
